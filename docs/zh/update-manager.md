# QBoot 升级接收流程框架

## 1. 模块定位

这个模块的定位是：**可配置实现内置升级接收流程框架**。

它不是强制组件，也不是唯一升级实现方式。它的价值在于把“等待接收、开始接收、接收中、接收完成、接收失败、恢复探测”这些通用状态统一起来，让协议层和存储层不必各自维护一套分散状态机。

## 2. 它负责什么

典型职责包括：

- 管理升级原因与运行状态
- 处理等待窗口与接收空闲超时
- 在 APP 不可用时触发恢复探测
- 统计接收进度
- 在 helper 模式下统一处理接收区打开、擦除和写入

## 3. 什么时候适合启用

适合以下情况：

- bootloader 上电后需要等待升级输入
- 需要区分“正常启动”和“请求升级启动”
- 需要统一接收状态和错误处理
- 需要在 APP 无效时尝试恢复或保底策略

## 4. 什么时候可以不启用

以下情况通常可以先不启用：

- 协议层已经有完整状态机
- bootloader 不需要等待窗口
- 升级输入路径非常简单，外部逻辑已经包办全部控制

## 5. 接入前提

### 5.1 启用配置项
- `QBOOT_USING_UPDATE_MGR`

### 5.2 提供回调集合
需要实现并注册 `qbt_update_ops_t`，常见回调包括：

- `is_app_valid`
- `get_reason`
- `set_reason`
- `enter_download`
- `leave_download`
- `on_error`
- `on_ready_to_app`
- `try_recover`

## 6. helper 模式与非 helper 模式

### 6.1 helper 模式
启用：
- `QBOOT_UPDATE_MGR_USE_DOWNLOAD_HELPER`

适合：
- 希望把接收目标区打开、擦除、写入统一交给组件处理
- 协议层只负责把数据流喂进来

### 6.2 非 helper 模式
适合：
- 你已有完整接收存储流程
- 你只想复用状态控制逻辑

## 7. download helper 的长度职责边界

`qbt_update_mgr_download_begin()`、`qbt_update_mgr_download_write()` 和
`qbt_update_mgr_download_finish()` 只是存储会话 helper。它们不跟踪完整
接收长度，不强制包顺序，不拒绝重复写、重叠写或空洞写，
也不比较传输层声明的总长度。

职责边界如下：

- 协议适配层负责传输层总长度、顺序、重试、重复、重叠和空洞策略；
- 存储后端负责写入范围和容量检查；
- 固件解析 / release 路径负责包级 `pkg_size`、`raw_size`、CRC、
  解压 / 解密输出长度以及目标分区容量校验。

调用 `qbt_update_mgr_download_finish(RT_TRUE)` 表示调用方已经接受当前
接收对象。helper 本身不会独立证明对象已经完整。对于 RBL 包，畸形包、
截断包、超大包、零长度包、CRC 不匹配以及目标容量不足等情况，会在后续
固件检查 / release 路径中被拒绝。当前策略允许声明包体之后存在额外尾部
数据，解析时按包内声明长度忽略这些额外数据。

## 8. 状态理解

典型可以理解为：

- **WAIT**：等待升级输入
- **RECV**：正在接收升级数据
- **READY**：已具备继续处理或切换应用的条件

## 9. 接入建议

1. 先确认你的升级输入入口是什么
2. 再决定是否需要 helper 模式
3. 把升级原因做持久化
4. 把恢复探测做成可重复验证且副作用可控
5. 先跑通接收，再考虑进度显示和日志优化

## 10. 和其他模块的关系

- 与自定义接收协议配合
- 与 custom/FAL/FS 后端配合
- 与 APP 校验和恢复逻辑共同决定何时进入 APP

## 11. 调试重点

优先检查：

- 升级原因读写是否可靠
- begin/write/finish 调用顺序是否正确
- 超时参数是否合理
- `try_recover()` 是否反映真实结果
- helper 模式下目标区擦写是否真实成功
